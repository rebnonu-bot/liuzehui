# Umami 统计查询性能与缓存策略文档

## 1. 整体架构

```
用户访问 → 多层级缓存 → 最终数据源
     ↓
[Browser] → [Cloudflare Edge] → [Origin Server] → [Umami Server]
   2分钟        6小时              6小时             实时
```

## 2. 三级缓存策略

### 第一层：客户端缓存（Browser）
**文件**: `src/hooks/use-article-hits.ts`

```typescript
const CACHE_TTL_MS = 2 * 60 * 1000; // 2 分钟
```

**特点**:
- 全局共享缓存（所有组件共用一份）
- 页面内切换不重复请求
- 刷新页面后缓存失效
- 站内外链跳转保留缓存

**使用场景**:
```
首页 → 文章详情 → 回到首页
   ↓
首次加载请求数据，后续 2 分钟内使用缓存
```

### 第二层：Edge 缓存（Cloudflare Workers）
**文件**: `src/app/api/analytics/hits/route.ts`

```typescript
export const revalidate = 21600; // 6 小时
const CACHE_TTL = 6 * 60 * 60 * 1000; // 6 小时内存缓存
```

**特点**:
- 全球 300+ 数据中心就近响应
- 大幅减少回源请求
- 缓存命中时响应时间 < 50ms
- 通过 `X-Cache: HIT/MISS` 头部可观察

**实际效果**:
```
用户 A 访问（北京）→ 回源获取 → 缓存到北京节点
用户 B 访问（北京）→ 直接命中缓存
用户 C 访问（上海）→ 可能回源或命中上海节点缓存
```

### 第三层：服务端内存缓存（Origin Server）
**文件**: `src/lib/content/listings.ts`

```typescript
const CACHE_TTL_MS = 6 * 60 * 60 * 1000; // 6 小时
```

**特点**:
- SSR 渲染时共享缓存
- 服务端渲染文章列表时使用
- 避免每个请求都调用内部 API

## 3. 数据流向详解

### 场景 1：用户首次访问首页（SSR）
```
[用户浏览器]
    ↓
[Cloudflare Edge] ──→ 检查 Edge Cache（未命中）
    ↓
[Origin Server] ──→ 检查 Server Cache（未命中）
    ↓
[Umami API] ──→ 获取数据（约 200-500ms）
    ↓
逐层返回并写入缓存
```

**实际耗时**: 首次 ~500ms，后续 Edge 命中 ~50ms

### 场景 2：站内文章跳转（CSR）
```
文章 A → 文章 B
   ↓
usePageHits() 检查 globalCache
   ↓
缓存有效期内（2分钟）→ 直接使用，0ms
缓存过期 → 请求 /api/analytics/hits
   ↓
Edge Cache 命中 → 直接返回
Edge Cache 过期 → 回源获取
```

**实际耗时**: 缓存命中 0ms，Edge 命中 ~50ms，回源 ~500ms

### 场景 3：返回首页（CSR）
```
文章详情 → 回到首页
   ↓
useArticleHits() 发现已有 globalCache
   ↓
直接复用缓存数据，无网络请求
```

**实际耗时**: 0ms

## 4. 会不会打到统计服务器？

### 直接访问 Umami 的情况
| 场景 | 频率 | 说明 |
|------|------|------|
| 6 小时内首个请求 | 每 6 小时 1 次 | Edge Cache 过期时 |
| 手动清除缓存 | 极低 | 重新部署或主动清除 |
| 缓存失效回源 | 每 6 小时 1 次 | 正常情况 |

### 数学计算
假设网站有 1000 UV/天：
```
无缓存：1000 次请求 × 10 页面 = 10,000 次/天打到 Umami
有缓存：24 小时 / 6 小时 = 4 次/天打到 Umami

节省：99.96% 的请求
```

## 5. 实时性问题

### 数据延迟
| 层级 | 延迟时间 | 说明 |
|------|---------|------|
| 用户看到的数据 | 最多 6 小时 | Edge Cache 决定 |
| 站内切换 | 最多 2 分钟 | Client Cache 决定 |
| 新文章统计 | 0 | 新数据下次缓存过期后显示 |

### 是否需要实时？
**不需要**，原因：
1. 历史统计数据变化频率低（每天变化）
2. 用户关注相对排名而非精确数字
3. 6 小时延迟对展示无影响

## 6. 缓存失效策略

### 自动失效
- Edge Cache: 6 小时后自动过期
- Client Cache: 2 分钟后自动过期
- Server Cache: 6 小时后自动过期

### 手动刷新（如需）
```bash
# 清除 Cloudflare 缓存
# 在 Cloudflare Dashboard 中清除

# 或重新部署
pnpm deploy:vinext
```

## 7. 性能监控

### 观察缓存命中率
查看响应头：
```
X-Cache: HIT    # Edge 缓存命中
X-Cache: MISS   # Edge 缓存未命中
X-Cache: STALE  # 使用过期的缓存（容错）
```

### 预期性能
| 指标 | 目标值 | 实际表现 |
|------|--------|---------|
| Edge 缓存命中率 | > 95% | 预期 99%+ |
| 平均响应时间 | < 100ms | Edge 命中 ~50ms |
| Umami 服务器负载 | 极低 | 每 6 小时 1-2 请求 |

## 8. 优化建议

### 当前配置已足够
- ✅ 三级缓存覆盖所有场景
- ✅ 6 小时缓存适合统计数据
- ✅ 容错机制完善（STALE 缓存）

### 如需进一步优化
1. **更长的 Edge Cache**: 将 `revalidate` 从 6 小时改为 24 小时
2. **预渲染**: 构建时生成静态数据
3. **数据预热**: 定时任务提前更新缓存

## 9. 总结

**Q: 用户访问会打到统计服务器吗？**
A: 极少。99%+ 的请求被 Cloudflare Edge 缓存拦截，只有每 6 小时的第一个请求会打到 Umami。

**Q: 站内切换有实时请求吗？**
A: 2 分钟内没有。客户端缓存共享，2 分钟后才会请求新数据。

**Q: 数据实时性如何？**
A: 最多 6 小时延迟，对历史浏览量展示完全可接受。

**Q: 性能如何？**
A: 极快。Edge 命中 < 50ms，客户端缓存 0ms。
